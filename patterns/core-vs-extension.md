# Core vs Extension

How do you decide what's an extension or what should be in core? You don't want to overload the user the user with a mountain of functionality, but underselling the possibilities is not the right decision either.

I think the key is to gradually introduce the options to the user of the software. The "gentle slope from user to creator" from the I&S malleable software [paper](https://www.inkandswitch.com/essay/malleable-software/#a-gentle-slope-from-user-to-creator) fits in nicely here.

**To determine what's in core you want to find a balance between essential features and a few additional features that demonstrate the range of what is possible.**


## 1st vs 3rd party extensions

Internal extensions are usually preferred because we trust that the software provider has fully verified their authenticity. It also tends to be on the safer side because they're not subject to external influences.

This could be made into a more democratic process. Take profile [verification](https://bsky.social/about/blog/04-21-2025-verification) on the AT Protocol as an example, verification is no longer centralised with one social media company; people can choose which verifiers they trust and consequently see a badge next to the profiles those verifiers have approved.

We could apply this to extensions, people could verify a particular version (range) of an extension. A combination of content addressing and social structures. Hopefully that way the line between 1st and 3rd party can fade a little.
